# Actions 模块需求

## 1. 背景

Codex App 已支持“环境 -> 操作 -> 添加操作”，并把环境动作持久化到项目内的 `.codex/environments/environment.toml`。

当前项目已经有一套成熟的移动端模块接入骨架：

- 顶部 `More` 菜单触发入口。
- `App` 内维护独立 panel state。
- `AppShell` 挂载独立 panel 组件。
- `server/*-routes.js` 承接 API。
- `server/*-service.js` 承接项目级业务逻辑。
- 全部能力都通过 `getProject(projectId)` 把执行边界限制在当前选中项目。

Git 模块已经采用这套骨架，因此本次最短路径不是复用 Git 业务，而是复用 Git 的接入方式，在移动端新增一个与 Git 并列的 `Actions` 模块。

## 2. 命名与兼容原则

- 移动端对用户暴露的模块名统一为 `Actions`。
- 兼容基准不是自定义新格式，而是桌面端真实持久化文件 `.codex/environments/environment.toml`。
- 本次需求明确以真实路径 `.codex/environments/environment.toml` 为准，不使用错误路径 `.codex/environmentsenvironment.toml`。
- 移动端只操作 `[[actions]]`。
- `[setup]` 必须完整保留，不允许因为移动端写回而丢失。
- 未识别字段必须尽量保留，不能因为移动端保存导致桌面端配置退化。

## 3. 当前已确认的桌面端文件结构

当前仓库中的真实样例如下：

```toml
# THIS IS AUTOGENERATED. DO NOT EDIT MANUALLY
version = 1
name = "CodexMobile"

[setup]
script = ""

[[actions]]
name = "构建"
icon = "run"
command = "npm run build"

[[actions]]
name = "运行并启动tailscale"
icon = "run"
command = '''
tailscale serve --bg 3321
pm2 start ecosystem.config.cjs
'''
platform = "win32"
```

由此可确认：

- 文件是项目级别配置，不是全局配置。
- `actions` 是 TOML 数组表。
- `command` 支持单行字符串和多行字符串。
- `platform` 是可选字段。
- 当前已观测字段至少有 `name`、`icon`、`command`、`platform`。

## 4. 目标

在 CodexMobile 中新增 `Actions` 模块，使用户可以在手机端：

- 查看当前项目 `.codex/environments/environment.toml` 内的全部 actions。
- 执行 action。
- 新增 action。
- 编辑 action。
- 删除 action。
- 感知 action 的平台限制、执行中状态、执行结果和执行失败原因。

同时满足：

- 与桌面端读取同一份配置。
- 不引入第二份动作存储。
- 不允许越出当前项目工作区执行。
- 不允许因为移动端写回破坏桌面端文件结构。

## 5. 非目标

- 本期不做多 environment 切换。
- 本期不做 action 拖拽排序。
- 本期不做 `[setup]` 的执行入口。
- 本期不做任意自由文本即时执行框。
- 本期不做后台常驻任务管理器。
- 本期不做浏览器自动验收；UI 验证由人工手动验证。

## 6. 用户故事

### 6.1 查看

用户在移动端选中某个项目后，可以从顶部 `More` 菜单进入 `Actions` 面板，看到：

- 环境名称。
- 配置文件路径。
- `[setup].script` 是否存在。
- actions 列表。

### 6.2 执行

用户点击某个 action 后，移动端先展示确认信息，再执行其 `command`。

确认信息至少包含：

- action 名称。
- 平台限制。
- 当前项目路径。
- 即将执行的完整命令文本。

### 6.3 新增

用户可以新增一个 action，至少填写：

- 名称。
- 图标。
- 命令。
- 平台限制，可选。

保存后应直接写入 `.codex/environments/environment.toml` 的 `[[actions]]`。

### 6.4 编辑

用户可以编辑已有 action 的：

- 名称。
- 图标。
- 命令。
- 平台限制。

### 6.5 删除

用户可以删除已有 action，删除前必须二次确认。

## 7. 核心约束

### 7.1 安全边界

- 所有 action 执行都只能以当前 `project.path` 作为 `cwd`。
- 不允许 action 自定义 `cwd` 到项目外。
- 不允许通过移动端传入新的自由命令文本直接执行。
- 只能执行已保存在 `.codex/environments/environment.toml` 中的 action。
- 所有执行都必须有前置确认。
- 服务端必须设置固定超时和输出上限。
- 服务端必须返回明确的退出码、stdout、stderr 和超时状态。

### 7.2 兼容边界

- 写回时必须保留文件头、`version`、`name`、`[setup]`。
- 未编辑的 action 不应因为重写而被语义修改。
- 多行命令写回后再次读取，内容必须保持等价。
- 如果文件缺失，允许按桌面端结构自动创建。
- 如果文件损坏或 TOML 非法，必须显式报错并停止，不能静默覆盖。

### 7.3 平台边界

- `platform` 为空时，表示全部平台可见可运行。
- `platform` 存在且与 `process.platform` 不一致时，默认展示为禁用态。
- 禁用态要明确提示“当前平台不可运行”。

## 8. 交互需求

### 8.1 顶部入口

在顶部 `More` 菜单中新增 `Actions` 区块或入口按钮。

要求：

- 与 `Git` 面板并列。
- 仅在存在 `selectedProject` 时可用。
- 当前有运行中的主线程时可继续查看。
- 是否允许运行 action 与是否允许打开面板分离。

### 8.2 Actions 面板

面板风格沿用 Git 面板现有移动端样式，不另起设计体系。

建议结构：

- 顶部标题区。
- 环境摘要卡片。
- action 列表区。
- 新增 action 按钮。
- 编辑 action 表单。
- 删除确认区。
- 运行确认区。
- 执行结果区。

### 8.3 Action 卡片

每个 action 卡片展示：

- 名称。
- 图标。
- 平台标签。
- 命令预览，超过长度折叠。
- 可执行状态。

卡片行为：

- 主操作为“运行”。
- 次操作为“编辑”和“删除”。

### 8.4 新增/编辑表单

表单字段：

- `name`：必填。
- `icon`：必填，先只支持现有已观测值和少量预设值。
- `command`：必填，多行文本框。
- `platform`：可选，枚举 `all`、`win32`、`darwin`、`linux`。

表单校验：

- `name` 不能为空。
- `command` 不能为空。
- `icon` 不能为空。
- `platform=all` 写回时不落盘该字段。

## 9. 服务端需求

### 9.1 文件解析与写回

新增独立服务模块，建议命名：

- `server/actions-service.js`
- `server/actions-routes.js`

服务端职责：

- 根据 `projectId` 解析目标文件路径：`<project.path>/.codex/environments/environment.toml`。
- 读取 TOML。
- 解析环境元数据与 actions。
- 执行 action。
- 写回新增、编辑、删除后的 TOML。

### 9.2 API 设计

建议最小 API：

- `GET /api/actions?projectId=...`
- `POST /api/actions/run`
- `POST /api/actions`
- `PATCH /api/actions`
- `DELETE /api/actions`

请求约束：

- 所有接口都必须要求 `projectId`。
- 变更类接口都必须携带 `revision`。
- 运行、编辑、删除必须携带 `actionKey`。

### 9.3 返回结构

`GET /api/actions` 返回：

- `environment`
- `revision`
- `exists`
- `path`

`environment` 至少包含：

- `version`
- `name`
- `setupScriptPresent`
- `setupScript`
- `actions`

`actions[]` 至少包含：

- `actionKey`
- `index`
- `name`
- `icon`
- `command`
- `platform`
- `platformMatched`

### 9.4 action 标识

当前桌面端文件未观测到 `id` 字段，因此本期不向 TOML 强塞新字段。

本期使用服务端生成的瞬时标识：

- `actionKey = index + 内容摘要`

变更时同时校验：

- `revision`
- `actionKey`

如果文件已被桌面端或其他端改动导致摘要不一致，直接报冲突并要求前端刷新。

### 9.5 执行模型

为了兼容当前 action 的 `command` 写法，执行不能走 `execFile(command, args)` 这种拆词模型，必须走 shell。

推荐执行方式：

- Windows：`powershell -NoProfile -NonInteractive -Command <command>`
- 非 Windows：`/bin/sh -lc <command>`

执行控制：

- `cwd` 固定为当前 `project.path`
- 默认超时 10 分钟
- 默认输出上限 1 MB
- 超时后强制结束并返回 `timedOut=true`
- 同一项目同一时刻只允许一个 action 运行

### 9.6 自动建档

如果 `<project.path>/.codex/environments/environment.toml` 不存在：

- `GET /api/actions` 返回空 actions 和 `exists=false`
- 当用户首次新增 action 时，服务端自动创建目录和文件

建议初始模板：

```toml
# THIS IS AUTOGENERATED. DO NOT EDIT MANUALLY
version = 1
name = "<project-name>"

[setup]
script = ""
```

## 10. 前端需求

### 10.1 状态接入

新增独立 panel state，参考 Git 面板接入方式。

建议修改点：

- `client/src/app/AppState.js`
- `client/src/app/App.jsx`
- `client/src/app/AppShell.jsx`
- `client/src/panels/index.js`
- `client/src/panels/TopBar.jsx`
- `client/src/panels/ActionsPanel.jsx`

### 10.2 前端行为

面板打开时：

- 读取当前项目 actions。
- 清空上次执行结果。
- 显示配置文件路径和环境名。

点击运行时：

- 弹出确认区。
- 用户确认后调用 `POST /api/actions/run`。
- 展示运行中状态。
- 运行完成后展示 stdout/stderr/exitCode。

新增、编辑、删除成功后：

- 重新拉取最新 environment。
- 更新 `revision`。
- 用 toast 给出结果提示。

### 10.3 错误展示

必须直接暴露的错误：

- 环境文件不存在。
- TOML 解析失败。
- 平台不匹配。
- action 内容为空。
- 修订冲突。
- 执行超时。
- 运行失败。

不得做静默降级或偷偷覆盖。

## 11. TOML 支撑方案

当前仓库未发现现成 TOML 依赖。

本需求推荐方案：

- 服务端新增轻量 TOML 读写依赖。
- 只在 `actions-service` 内部使用。
- 读写结果要覆盖当前桌面端文件结构。

不推荐方案：

- 手写脆弱正则直接改字符串。

如果因为依赖策略不能新增 TOML 库，则最小可接受降级为：

- 只支持当前已知结构。
- 对未知字段做保守透传。
- 文档中明确“未来桌面端格式变更时有兼容风险”。

## 12. 风险与规避

### 12.1 命令危险性高

风险：

- action 可执行破坏性命令。

规避：

- 只允许运行已保存 action。
- 每次运行前显式确认。
- 展示完整命令。

### 12.2 文件并发修改

风险：

- 桌面端与移动端同时编辑同一份 `environment.toml`。

规避：

- 使用 `revision + actionKey` 冲突检测。
- 冲突后停止写入并提示刷新。

### 12.3 平台差异

风险：

- Windows action 在 macOS/Linux 上误运行。

规避：

- `platform` 不匹配直接禁用。

### 12.4 多行命令兼容

风险：

- 写回后桌面端无法正确解析。

规避：

- 测试覆盖单行、多行、空 setup、带 platform 的 action 写回。

## 13. 验收标准

满足以下条件即视为本需求完成：

- 进入移动端 `Actions` 面板时，能读取当前项目 `.codex/environments/environment.toml`。
- 样例中的 3 个 actions 都能被正确展示。
- `platform = "win32"` 的 action 在 Windows 下可运行，在其他平台展示禁用原因。
- 新增 action 后，桌面端重新读取同一文件能看到新增项。
- 编辑 action 后，桌面端重新读取同一文件能看到更新项。
- 删除 action 后，桌面端重新读取同一文件看不到该项。
- `[setup]` 区块在多次新增、编辑、删除后仍保留。
- 非法 TOML 会报错并阻止写回。
- action 执行结果能返回退出码与输出摘要。

## 14. 测试要求

至少补充：

- `server/actions-service.test.mjs`
- `server/actions-routes.test.mjs`
- 与前端请求映射相关的 `client/src/*actions*.test.mjs`
- 受影响 UI 状态的最小单测

至少执行：

- `node --test server/*.test.mjs client/src/*.test.mjs shared/*.test.mjs`
- `npm run build`

页面级联调和交互验收本次不使用 Browser 插件，需人工手动验证。

## 15. 实施顺序

建议按最小路径分三步：

### Phase 1

- 打通 `GET /api/actions`
- 前端面板只读展示
- 支持运行 action

### Phase 2

- 支持新增、编辑、删除 action
- 补 revision 冲突保护

### Phase 3

- 优化图标、平台标签、输出展示
- 增加首次无文件初始化体验

## 16. 结论

本需求可以做，而且最短最稳的实现方式已经明确：

- 模块对外命名为 `Actions`
- 底层直接兼容 `.codex/environments/environment.toml`
- 不新建第二份动作存储
- 只围绕 `[[actions]]` 做读取、执行和编辑
- 通过项目级 cwd、显式确认、超时、输出上限和冲突检测把风险收住
